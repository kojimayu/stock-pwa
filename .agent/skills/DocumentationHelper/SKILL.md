# DocumentationHelper Skill

このスキルは、プロジェクトのドキュメント管理ルールとワークフローを定義します。

## ⚠️ 重要ルール (Critical Rules)

1. **言語**: ドキュメントは**常に日本語**で記述してください。
2. **Source of Truth**: `docs/` 内のドキュメントを、常にプロジェクトの進捗と設計の「信頼できる唯一の情報源」として維持してください。

## 管理対象ドキュメント (Managed Documents)

### 恒久ドキュメント（2ファイル）

1. **`docs/FEEDBACK.md`** — プロジェクトのマスターバックログ
   - **目的**: 未対応タスク・バグ・改善要望・対応済み項目の一元管理
   - **更新タイミング**: バグ発見時、機能要望追加時、修正完了時
   - **形式**: 優先度別セクション（🔴最優先 / 🟠優先 / 🟡機能追加 / 🟢将来）+ 対応済みセクション

2. **`docs/CHANGELOG.md`** — リリース変更履歴
   - **目的**: コミットごとの変更内容の記録
   - **更新タイミング**: コミット前（必須）
   - **形式**: `### Fixed / Added / Changed` セクション、日付付き

### セッション中メモ（一時的）

- コミットまでの作業進捗はAntigravityの **artifact領域**（`brain/` 配下）に一時保存
- コミット時に `FEEDBACK.md` と `CHANGELOG.md` に反映
- セッション終了後は自動破棄される

> **注意**: `docs/<project_name>/task.md`, `implementation_plan.md`, `walkthrough.md` は**廃止**。新規作成しないこと。

## 運用ワークフロー (Workflow)

1. **新規タスク開始時**
   - `FEEDBACK.md` の該当項目を確認
   - 複雑な変更の場合、ユーザーに設計方針を事前説明し承認を得る

2. **⚠️ 承認取得（必須）**
   - **コードを編集する前に、変更内容をユーザーに説明し承認を得てください。**
   - 承認なしにコード変更を行うことは禁止です。
   - 単純なバグ修正（1ファイル・数行）の場合は口頭承認でOK。

3. **⚠️ テスト影響チェック（実装完了時に必須）**
   以下のファイルを変更した場合、**対応するテストの更新・追加が必要です**:
   - `lib/actions.ts` → `__tests__/actions/` 配下のテスト
   - 新しいサーバーアクション関数を追加した場合 → 新しいテストケースを作成
   - 既存関数の引数・戻り値を変更した場合 → 既存テストを更新
   - テスト更新後は `npm test` で全テストが通過することを確認してください。

4. **コミット前（必須）**
   - `docs/CHANGELOG.md` にエントリを追加
   - `docs/FEEDBACK.md` の該当項目を修正済みに更新
   - DBバックアップを取得（コード変更のみでも必須）

## 例外 (Exceptions)
- 単純なバグ修正や一般的な質問への回答では、ドキュメント更新を省略可。ただし**コード変更前の承認は常に必要**。
- UIのみの変更（CSS・レイアウト修正）では、テスト更新は不要。
