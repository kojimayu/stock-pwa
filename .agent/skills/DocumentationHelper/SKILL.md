# DocumentationHelper Skill

このスキルは、プロジェクトのドキュメント管理ルールとワークフローを定義します。
複雑な機能実装やプロジェクトの初期化を行う際は、必ずこのスキルに従ってください。

## ⚠️ 重要ルール (Critical Rules)

1. **言語**: ドキュメントは**常に日本語**で記述してください。
2. **Source of Truth**: `docs/` 内のドキュメントを、常にプロジェクトの進捗と設計の「信頼できる唯一の情報源」として維持してください。

## 管理対象ドキュメント (Managed Documents)

以下の3つのドキュメントを `docs/<project_name>/` ディレクトリ内で管理します。

1. **`task.md`**
   - **目的**: タスク一覧と進捗状況の管理
   - **更新タイミング**: タスクの開始時、完了時、ステータス変更時
   - **形式**: チェックボックス付きリスト (`- [ ]`, `- [/]`, `- [x]`)

2. **`implementation_plan.md`**
   - **目的**: 実装の詳細なステップと設計方針の定義
   - **更新タイミング**: 実装前（計画）、設計変更時
   - **内容**: 変更するファイル、具体的なロジック、検証計画

3. **`walkthrough.md`**
   - **目的**: 実施した変更内容の記録と意図の説明
   - **更新タイミング**: 実装完了後、PR作成前
   - **内容**: 変更点の概要、検証結果、スクリーンショット（あれば）

## 運用ワークフロー (Workflow)

1. **新規タスク開始時**
   - `task.md` にタスクを追加するか、既存タスクを進行中 (`[/]`) にマークします。
   - `implementation_plan.md` を作成または更新し、設計方針を明確にします。

2. **⚠️ 承認取得（必須）**
   - **コードを編集する前に、必ず `implementation_plan.md` をユーザーに提示し、承認を得てください。**
   - 承認なしにコード変更を行うことは禁止です。
   - 単純なバグ修正（1ファイル・数行の修正）の場合は、変更内容の概要をユーザーに説明し、口頭承認を得てから実施してください。

3. **実装中**
   - 必要に応じて `implementation_plan.md` を修正し、現状と計画の乖離を防ぎます。

4. **タスク完了時**
   - `walkthrough.md` を作成し、何を行ったかを記録します。
   - `task.md` の対象タスクを完了 (`[x]`) にマークします。

## 例外 (Exceptions)
- 単純なバグ修正や一般的な質問への回答では、ドキュメント作成を省略しても構いませんが、**コード変更前の承認は常に必要です**。
