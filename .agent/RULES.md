# プロジェクト固有ルール (stock-pwa)

## データベース操作

### マイグレーション実行時の安全確認
`prisma migrate dev` や `prisma migrate deploy` を実行する前に、**必ず以下を行うこと**：

1. `npm run start` や `npm run dev` が動いていないか確認する
2. 動いている場合は、**ユーザーに停止の承認を取ってから**停止する
3. マイグレーション完了後に、ユーザーにサーバー再起動を案内する

> **理由:** マイグレーション中にDBロックが発生し、同時にアクセスしているサーバーが固まる事故が発生したため（2026-02-21）。

### DATABASE_URLの確認
マイグレーション実行時は `DATABASE_URL` が本番DB（`file:./prisma/dev.db`）を指していることを必ず確認する。
テスト用DB（`file:./prisma/test-vitest.db`）に対してマイグレーションを実行しないこと。

### 破壊的操作前のバックアップ（必須）
以下の操作を行う**前に**、必ず `.\scripts\backup_db.ps1 -BackupFolder .\backups` を実行してバックアップを取ること：

- マイグレーション（`prisma migrate dev` / `prisma migrate deploy`）
- データの削除（DELETE文、テーブル全削除など）
- テーブル構造の変更（カラム追加・削除・型変更）
- スクリプトによるデータ一括更新

バックアップ先とファイル名をユーザーに明示し、復元方法もセットで報告すること。

> **理由:** Gitコミット時のバックアップだけでは、変更途中でデータが失われた場合に復元できない。変更前にバックアップを取ることで、いつでも直前の状態に戻せる（2026-02-25）。

## ドキュメント更新

### SYSTEM_UPDATES（ユーザー向け変更履歴）
機能追加・不具合修正・UI変更があった場合、`views/dashboard.py` の `SYSTEM_UPDATES` リストにエントリを追加すること。
※ 本プロジェクトはNext.js（TSX）のため、該当する対応箇所がある場合のみ。

### テスト・ドキュメント・マニュアルの同時更新
機能の変更や追加を行った場合は、**必ず以下の3点をセットで更新**すること：

1. **テスト**: 変更した機能に対応するテストを変更・追加する
2. **ドキュメント**: 以下を更新する
   - `docs/CHANGELOG.md` — 変更履歴
   - `docs/TESTING.md` — テスト一覧
   - `docs/FEEDBACK.md` — 対応済み項目の移動
3. **マニュアル**: UI・操作フローに変更がある場合は必ず更新する
   - `docs/MANUAL_ADMIN.md` — 管理者向けマニュアル
   - `docs/MANUAL_VENDOR.md` — 業者（Kiosk）向けマニュアル

> **理由:** 機能変更だけ先行してマニュアルやテストが古いまま放置されると、運用上の混乱やバグの見逃しにつながるため（2026-02-25）。
